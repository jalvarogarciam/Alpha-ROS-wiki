Entendiendo la arquitectura y los componentes fundamentales de ROS

Conceptos básicos en ROS

Para poder trabajar con ROS, es crucial comprender su arquitectura y los conceptos clave que permiten la comunicación entre las diferentes partes de un robot. ROS se ejecuta como un sistema distribuido sobre una red LAN.

El Núcleo de ROS: Master, Roscore y Rosout

  • Master (Nodo Máster): Imagina que es el "centro de control" o la "centralita telefónica" de ROS. Su función es permitir que los demás nodos se encuentren y se comuniquen entre ellos. Antes de ejecutar cualquier nodo de tu robot, el Master debe estar en funcionamiento.
  • roscore: Es el comando que lanza todo lo necesario para que el sistema base de ROS funcione. Al ejecutar roscore, se inician tres procesos:
    • El Master.
    • Rosout: un nodo que recoge los mensajes de log y los errores de los demás nodos.
    • El Servidor de Parámetros, una base de datos para la configuración.

Nodos (Nodes)

Un nodo es, en esencia, un programa. Es la unidad ejecutable en ROS. Cada nodo se diseña para realizar una tarea específica y concreta. Un sistema robótico complejo se construye a partir de la colaboración de muchos nodos pequeños y especializados.

Ejemplo: En el Alphabot2, un nodo podría dedicarse exclusivamente a leer los sensores de infrarrojos. Otro nodo se encargaría de controlar los motores. Un tercer nodo podría tomar las lecturas de los sensores (del primer nodo) y decidir a qué velocidad mover los motores (enviando órdenes al segundo nodo).

Topics y Messages: Comunicación Continua

  • Topics: Son los canales o "buses" de comunicación a través de los cuales los nodos intercambian información. Los nodos pueden publicar mensajes en un topic o suscribirse a un topic para recibir los mensajes que otros publican. La comunicación es asíncrona: el que publica no espera una respuesta.
  • Messages: Son los paquetes de datos que viajan a través de los topics. Cada topic está asociado a un tipo de mensaje específico (un entero, un texto, una imagen, etc.).

Analogía: Un topic es como un canal de televisión. Un nodo publicador es la emisora que transmite continuamente un programa (los mensajes), y los nodos suscriptores son los televisores que, al sintonizar ese canal, reciben el programa.

Servicios (Services): Comunicación por Petición

Servicios: Son otro mecanismo de comunicación, pero este es síncrono y se basa en un modelo de petición/respuesta (request/reply).

Funcionamiento: Un nodo (el cliente) llama a un servicio ofrecido por otro nodo (el servidor) y espera bloqueado hasta que el servidor procesa la petición y le devuelve una respuesta. Es una comunicación bidireccional y es ideal para acciones puntuales.

Ejemplo: Un nodo podría ofrecer un servicio llamado /reset_odometry. Otro nodo podría llamar a este servicio para poner a cero el contador de distancia recorrida del robot.

Topic vs Service

Característica Topic Servicio
Modelo de comunicación Publicar/Suscribir (Asíncrono) Petición/Respuesta (Síncrono)
Flujo de datos Continuo o periódico Puntual o bajo demanda
Dirección Unidireccional Bidireccional
Uso típico Flujo de datos de sensores, estado del robot Ejecutar comandos, consultar datos específicos

Herramientas Avanzadas: Bags y Launch Files

  • Bag files (.bag) : Son ficheros que permiten grabar todos los mensajes que se publican en uno o varios topics. Son extremadamente útiles para capturar los datos de un experimento y poder reproducirlos más tarde para depurar algoritmos sin necesidad de usar el robot real.
  • Launch files (.launch) : Son ficheros de configuración en formato XML que permiten lanzar y configurar múltiples nodos con un solo comando. Son imprescindibles para "arrancar" sistemas robóticos complejos, ya que automatizan el proceso de iniciar todos los componentes de software necesarios.
